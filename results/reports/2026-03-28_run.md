# Run Report: NL2SQL LoRA on Qwen2.5 (2026-03-28)

This report documents a single end-to-end run: dataset -> LoRA fine-tune -> execution-based evaluation -> base-vs-LoRA comparison.

## What We Built

- A minimal LoRA-only training script for ChatML-style datasets: `finetune_nl2sql/train_lora.py`
- An execution-match evaluator that routes examples to schema-specific seeded databases via a validator API: `finetune_nl2sql/eval_exec.py`
- One-command runners for Runpod: `finetune_nl2sql/run_*.sh`

## Dataset

- Train: `562` rows
- Dev: `103` rows
- Test: `90` rows
- Format: one JSONL row per example, with `messages[] = [system, user, assistant]`
  - System: `"Return only SQL"`
  - User: schema + business rules + question
  - Assistant: SQL (gold)

Note: the test set spans multiple schemas/domains, so the per-schema training signal is small.

## Metric: Execution-Match

Each test example is evaluated as:

1. Generate SQL from the model.
2. Execute both predicted SQL and gold SQL against the seeded SQLite DB for that schema.
3. Compare result hashes (`result_hash(pred) == result_hash(gold)`).

This is stricter than “answers the question” because **column aliases and column order can change the hash**.

## Models & Training Setup

**Qwen2.5-14B-Instruct (LoRA)**

- LoRA: `r=16`, `alpha=32`, `dropout=0.05`
- Target modules: `q_proj,k_proj,v_proj,o_proj,gate_proj,up_proj,down_proj`
- Train: `epochs=3`, `lr=2e-4`, `max_seq_len=1024`, packing enabled

**Qwen2.5-Coder-32B-Instruct (LoRA)**

- LoRA: `r=8`, `alpha=32`, `dropout=0.05`
- Train: `epochs=2`, `lr=1e-4`, `max_seq_len=1024`, packing enabled

## Example Improvements (14B Base -> 14B LoRA)

These are cases where base failed execution-match and LoRA passed.

### b2b_saas

Question: *Which active organizations had more than one paid invoice?*

Base (misses ordering / output contract):

```sql
SELECT o.id, o.name FROM organizations o JOIN invoices i ON o.id = i.organization_id
WHERE o.status = 'active' AND i.status = 'paid'
GROUP BY o.id, o.name HAVING COUNT(DISTINCT i.id) > 1
```

LoRA (matches the contract more closely; passes execution-match):

```sql
SELECT o.name, COUNT(i.id) AS paid_invoice_count
FROM organizations o JOIN invoices i ON i.organization_id = o.id
WHERE o.status = 'active' AND i.status = 'paid'
GROUP BY o.id, o.name
HAVING COUNT(i.id) > 1
ORDER BY paid_invoice_count DESC, o.name ASC
```

### logistics_fleet

Question: *Which origin warehouses have more than one shipment on active routes?*

LoRA produced the full aggregation + ordering and matched execution results.

## A Honest Regression (14B)

Question: *Show the month-over-month application count trend.*

Gold:

```sql
SELECT strftime('%Y-%m', submitted_at) AS month, COUNT(*) AS application_count
FROM applications
GROUP BY strftime('%Y-%m', submitted_at)
ORDER BY month ASC
```

Base passed, LoRA failed execution-match, despite being semantically similar:

```sql
SELECT strftime('%Y-%m', submitted_at) AS submission_month, COUNT(*) AS application_count
FROM applications
GROUP BY strftime('%Y-%m', submitted_at)
ORDER BY submission_month ASC
```

This illustrates the strictness of the metric: **different column aliases can be treated as different results** by hash-based evaluation.

## Takeaways

- LoRA improved execution-match for both models, but absolute accuracy stayed low on this multi-domain benchmark.
- The gains were not proportional to model size; 14B benefited more than 32B (relative).
- Next step for a “wow” score is to **specialize**:
  - Train/eval on a single schema
  - Increase rows per schema to `800-3000` train examples
  - Tighten the prompt to enforce output contract (column names/order)
